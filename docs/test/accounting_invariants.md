# Accounting Invariants

> These invariants hold in every state of the system. They are tested as properties (Hypothesis) and asserted on every state-changing test.

## I1 — Every journal entry balances

For every row in `journal_entries`:

```
sum(journal_lines.amount where direction='DR') == sum(journal_lines.amount where direction='CR')
```

For a multi-currency entry, balance check is against `base_amount` (tenant base). Foreign-currency residuals (max 0.01 in base) are absorbed into `7900 Realized FX Gain/Loss` by `accounting_guardian` — no other slack.

Property test (`backend/tests/property/test_journal_balance.py`):

```python
from decimal import Decimal
from hypothesis import given, strategies as st

@st.composite
def balanced_journal(draw):
    n_lines = draw(st.integers(min_value=2, max_value=20))
    debits = [draw(st.decimals(min_value=Decimal("0.01"), max_value=Decimal("100000.00"), places=2))
              for _ in range(n_lines // 2)]
    credits_total = sum(debits)
    # Generate credits that exactly equal debits in count and sum
    credits = draw(split_amount_strategy(credits_total, count=n_lines - len(debits)))
    return list(zip(["DR"] * len(debits) + ["CR"] * len(credits), debits + credits))

@given(balanced_journal())
def test_journal_posts_when_balanced(lines):
    je = post_journal(lines, tenant_id=TEST_TENANT)
    assert sum(l.amount for l in je.lines if l.direction == "DR") == \
           sum(l.amount for l in je.lines if l.direction == "CR")
```

## I2 — Posted journals are immutable

For any `journal_entries` row with `status='posted'`:

- No UPDATE to `journal_entries` or `journal_lines` succeeds (database trigger or row-level policy).
- Corrections occur via a new entry with `entry_type='reversing'` referencing the original.

## I3 — Tenant-scoped totals reconcile

For every tenant T:

- `sum(invoices.total where status in ('approved', 'sent', 'paid'))` ==
  `sum(journal_lines.amount where account='1200 AR' and direction='DR')` —
  `sum(payments.amount)`
- `sum(bills.total where status in ('approved', 'paid'))` ==
  `sum(journal_lines.amount where account='2000 AP' and direction='CR')` —
  `sum(bill_payments.amount where status='settled')`

These are asserted at the end of every e2e run.

## I4 — Period lock is total

If `period_locks` has a row for `(tenant_id, period)`:

- No INSERT into `journal_entries` with `entry_date` in that period succeeds.
- No UPDATE that would re-date an existing entry into that period succeeds.
- `accounting_guardian` agent rejects any such proposal at draft time.

## I5 — FX rate frozen at post time

Every `journal_lines` row stores both `amount` (foreign) and `base_amount` (tenant base).

- `base_amount = amount × fx_rate(foreign_currency → base_currency, journal.entry_date)`
- The rate used is captured in `journal_lines.fx_rate_id` (FK to `fx_rates` row at time of post).
- `fx_rates` rows are immutable; FX rate corrections create new rows, never overwrite.
- Subsequent rate changes do NOT retroactively adjust posted journals.

## I6 — Money precision preserved

- `Decimal('10.10') + Decimal('20.20') == Decimal('30.30')` — no float drift.
- All money inputs validated to ≤ 2 decimal places before any computation.
- API responses serialize money as strings (`"30.30"`, not `30.3`).

Property test:

```python
@given(
    st.decimals(min_value=0, max_value=Decimal("999999.99"), places=2),
    st.decimals(min_value=0, max_value=Decimal("999999.99"), places=2),
)
def test_money_addition_preserves_precision(a, b):
    result = a + b
    assert result.as_tuple().exponent == -2  # exactly 2 decimal places
```

## I7 — Idempotency on external events

For every `webhook_events` row:

- A second event with the same provider `event.id` does NOT create a second `payments` / `bill_payments` row.
- The second delivery is recorded with `replay=true` flag.

## I8 — Sub-ledger ↔ GL coupling

Every sub-ledger event produces exactly one journal entry, in the same transaction:

- DB trigger or service-layer wrapping guarantees this.
- Orphan journal entries (no sub-ledger source) only allowed for owner-created manual entries with audit trail.
- Orphan sub-ledger events (no journal) are a P0 alert.

## I9 — Invoice numbers monotonic per tenant

`invoices.invoice_number` is generated by a DB sequence per tenant; never application-generated.

- No two invoices in the same tenant share a number.
- Sequence does not skip on rollback (uses a counter table, not `SERIAL`/`IDENTITY`).

## I10 — Reporting numbers come from data, not the LLM

Any report rendered by `reporting_agent`:

- Numeric values are produced by tool calls returning structured data (not the LLM's text generation).
- The LLM may summarize, label, or explain — but every number in the output must be traceable to a specific tool result row.
- Eval: hallucinated number = test failure with severity P0.

## I11 — RLS enforcement is unconditional

For every tenant-scoped table:

- Direct DB connection without `app.current_tenant_id` set returns 0 rows (RLS denies).
- Service-role connection is only used inside the auth bootstrap and admin endpoints — never inside ordinary request handlers.

## I12 — Reversal preserves audit chain

A reversing entry:

- Has `original_entry_id` FK to the entry it reverses.
- Has opposite debit/credit pattern for the same accounts and amounts (base currency).
- Both entries remain visible; neither is deleted.
- Audit log records the user, reason, and timestamp of the reversal.

## I13 — Autonomy demotion is sticky

When `autonomy_promoter_worker` demotes an agent from L3 to L2:

- The demotion is logged with reason (e.g., approval rate dropped).
- Re-promotion requires meeting the eligibility thresholds again AND explicit admin approval.
- The 90-day cooldown applies before the worker proposes re-promotion.
